# SOP — dbt on Amazon Redshift Serverless (Silver Iceberg → Gold native)

**Version:** 1.1 · **Last-reviewed:** 2026-06-17 · **Status:** Active
**R4 update (2026-06-17, F-AFIE-14):** §14 Pitfalls table gains explicit row codifying the AFIE F-FIN-05 retro: the upstream workgroup partials this composite consumes (MLOPS_DATA_PLATFORM, DATA_LAKEHOUSE_ICEBERG, DATA_ZERO_ETL) MUST set `max_capacity` explicitly. Without it the workgroup auto-scales to 512 RPU with no ceiling — $300+/hr burn during a runaway dbt MERGE.
**Applies to:** dbt-core ≥ 1.8 (or dbt Cloud) · `dbt-redshift` adapter ≥ 1.8 · Amazon Redshift Serverless · Silver in S3 Tables (Iceberg via Glue federated catalog) · Gold in Redshift native (RMS columnar)

**Validated against AWS docs 2026-05-12.** Redshift Serverless reads S3 Tables via federated `s3tablescatalog` external schema with no Spectrum surcharge.

---

## 1. Purpose

Codify how the team uses **dbt** to materialise the **Gold layer in Redshift Serverless native storage** from Silver Iceberg tables in S3 Tables. This is the AWS-recommended path when:

- The primary consumer is **Power BI / Tableau / QuickSight** (BI hot path).
- Latency target is **sub-second dashboards**.
- Silver lives in S3 Tables (open Iceberg, multi-engine-shareable).
- Gold is a **closed, BI-optimised** datastore — not intended for Athena / EMR / SageMaker direct access.

If Gold must remain Iceberg (multi-engine consumers), use `data/14` for Silver→Gold instead and skip this partial.

When the SOW signals: "Power BI is the consumer", "single source of truth", "dashboard latency matters", "team has SQL skills" → **use dbt for Silver→Gold**.

---

## 2. Why dbt and not Glue for Silver→Gold

| Concern | Glue (PySpark) | **dbt (SQL)** |
|---|---|---|
| Language for SQL-shaped transforms | PySpark DataFrame API or `spark.sql()` | Pure SQL — natural fit |
| Lineage | DDB lineage logger (custom) | `manifest.json` + `catalog.json` (free) |
| Tests | pytest + chispa (custom) | `dbt test` (declarative, in-repo) |
| Incremental materialisation | hand-rolled MERGE | `is_incremental()` + unique_key, ~5 lines |
| Cost | Glue DPU + Redshift COPY | Redshift CTAS / MERGE only |
| Analyst contribution | Python skill required | SQL only — much wider talent pool |
| CI integration | `pytest src/glue/tests/` | `dbt test` + `dbt parse` |

**dbt wins for Silver→Gold modelling on Redshift.** Glue stays for Bronze→Silver (where the transforms are Python-shaped, not SQL-shaped).

## 6. Example Gold model — `fct_unified_sales`

```sql
-- models/marts/core/fct_unified_sales.sql
{{
  config(
    materialized='incremental',
    unique_key='transaction_id',
    on_schema_change='fail',
    incremental_strategy='merge',
    dist='store_sk',
    sort=['transaction_date'],
    pre_hook="ANALYZE PREDICATE COLUMNS fct_unified_sales"
  )
}}

WITH src_ncr AS (
    SELECT
        transaction_id,
        transaction_date,
        store_id,
        customer_id,
        article_code,
        amount AS net_sales,
        'ncr' AS source,
        _run_id AS _source_run_id,
        _ingested_at AS _source_ingested_at
    FROM {{ source('silver', 'ncr_sales') }}
    {% if is_incremental() %}
      WHERE _ingested_at > (SELECT MAX(_source_ingested_at) FROM {{ this }})
    {% endif %}
),

src_ecom AS (
    SELECT
        order_id AS transaction_id,
        order_date AS transaction_date,
        NULL::INTEGER AS store_id,
        customer_id,
        article_code,
        net_amount AS net_sales,
        'ecommerce' AS source,
        _run_id AS _source_run_id,
        _ingested_at AS _source_ingested_at
    FROM {{ source('silver', 'ecommerce_orders') }}
    {% if is_incremental() %}
      WHERE _ingested_at > (SELECT MAX(_source_ingested_at) FROM {{ this }})
    {% endif %}
),

combined AS (
    SELECT * FROM src_ncr
    UNION ALL
    SELECT * FROM src_ecom
),

enriched AS (
    SELECT
        c.transaction_id,
        c.transaction_date,
        d_store.store_sk,
        c.customer_id,
        d_cat.category_sk,
        d_cat.category_name,
        a.brand,
        c.net_sales,
        c.source,
        c._source_run_id,
        c._source_ingested_at
    FROM combined c
        LEFT JOIN {{ ref('dim_store') }} d_store ON d_store.store_id = c.store_id
        LEFT JOIN {{ source('silver', 'sap_articles') }} a ON a.article_code = c.article_code
        LEFT JOIN {{ ref('dim_category') }} d_cat ON d_cat.category_id = a.category_id
)

SELECT
    {{ audit_columns() }},
    *
FROM enriched
```

## 6.5 Scenario / version-dimension pattern (added 2026-05-19)

Many retail / finance BI surfaces think in **multiple scenarios** per business-key grain rather than single-row facts. Tamimi's Power BI carries 3 scenarios per `Site × Date × Dept`:

- `"2026 A"` — current-year **A**ctuals
- `"2025 A"` — prior-year **A**ctuals (PY)
- `"Budget"` — budget plan

DAX measures filter on the scenario column (`CALCULATE(SUM(Sale), [Year]="2026 A")`) — so the Gold table must materialise all 3 scenarios as rows, not as columns.

### When to use a scenario column

- The same business key needs side-by-side comparison across versions (CY vs PY vs Budget)
- Power BI / Tableau / Looker measures slice by scenario as a filter
- New scenarios get added periodically (Forecast, Rolling-90-day, etc.) — column-per-scenario doesn't scale

### When NOT to use a scenario column

- Only one scenario exists today (just CY actuals) — don't over-engineer
- The "scenarios" are time-series snapshots — use SCD2 with `valid_from`/`valid_to` instead
- The scenarios have different schemas (Budget has fewer cols than Actuals) — keep them as separate tables

### dbt model shape — union of scenarios

```sql
-- models/marts/gold/unified_sales.sql
{{ config(
    materialized='incremental',
    incremental_strategy='merge',
    unique_key=['site', 'date', 'dept', 'scenario'],
    on_schema_change='fail',
    dist='date', sort=['date', 'site']
) }}

WITH actuals_cy AS (
    SELECT site, date, dept, '2026 A'::varchar AS scenario,
           SUM(sale) AS sale, SUM(cc) AS cc
    FROM {{ ref('silver_sap_zsdcc_enriched') }}
    WHERE EXTRACT(YEAR FROM date) = 2026
    GROUP BY 1,2,3,4
),

actuals_py AS (
    SELECT site, date, dept, '2025 A'::varchar AS scenario,
           SUM(sale) AS sale, SUM(cc) AS cc
    FROM {{ ref('silver_sap_zsdcc_enriched') }}
    WHERE EXTRACT(YEAR FROM date) = 2025
    GROUP BY 1,2,3,4
),

budget AS (
    SELECT site, date, dept, 'Budget'::varchar AS scenario,
           sale_budget AS sale, cc_budget AS cc
    FROM {{ ref('silver_budget_upload') }}
    -- Budget source resolution — see PATTERN_DDB_CONTROL_PLANE.source_catalog
)

SELECT * FROM actuals_cy
UNION ALL
SELECT * FROM actuals_py
UNION ALL
SELECT * FROM budget

{% if is_incremental() %}
WHERE date >= (SELECT MAX(date) - INTERVAL '7 days' FROM {{ this }})
{% endif %}
```

### Composite incremental key — critical detail

For scenario tables, `unique_key` MUST include `scenario`. If you forget, the `merge` overwrites actuals with budget (whichever scenario sorted last). The grain is `(site, date, dept, scenario)` — model accordingly.

### Reporting views over a scenario fact

DAX-translation views consume the scenario column natively:

```sql
-- models/marts/reporting/vw_sales_actuals_pyc.sql  (push-down of 4 DAX measures)
{{ config(materialized='view') }}

SELECT
    site, date, dept,
    SUM(sale) FILTER (WHERE scenario = '2026 A')               AS sale_cy_actual,
    SUM(sale) FILTER (WHERE scenario = '2025 A')               AS sale_py_actual,
    SUM(sale) FILTER (WHERE scenario = 'Budget')               AS sale_budget,
    SUM(sale) FILTER (WHERE scenario = '2026 A') -
        SUM(sale) FILTER (WHERE scenario = 'Budget')           AS sale_var_vs_budget,
    (SUM(sale) FILTER (WHERE scenario = '2026 A') -
     SUM(sale) FILTER (WHERE scenario = 'Budget')) /
     NULLIF(SUM(sale) FILTER (WHERE scenario = 'Budget'), 0)   AS sale_var_vs_budget_pct
FROM {{ ref('unified_sales') }}
GROUP BY site, date, dept
```

This replaces 5+ DAX measures with one SELECT. Power BI reads the view; DAX becomes a simple column reference (`SUM(sale_cy_actual)`), not a `CALCULATE`.

### Budget source — operational consideration

Budget rows come from somewhere that's NOT the transactional source (SAP / NCR). Options, in order of preference:

1. **Customer's planning system** (SAP IBP, SAP Analytics Cloud, Anaplan) — pull via a separate Bronze entity (`bronze.planning.budget`). Register as a `source_catalog` row.
2. **Excel / CSV upload** — drop into S3 landing prefix; `s3_landing` connector (see `PATTERN_SOURCE_AGNOSTIC_INGESTION` §4.4) picks it up.
3. **Customer-edited DDB table** — for small budgets edited via the operator UI (see `PATTERN_CUSTOMER_MAINTAINED_DIM_DDB`).

Decide at engagement W1; don't hardcode the Budget source in the dbt model — `{{ ref('silver_budget_upload') }}` is a stable reference.

## 7. Reporting view — the "move down from Power BI" layer

```sql
-- models/marts/reporting/vw_customer_count_dedup.sql
{{ config(materialized='view') }}

-- Replaces Amjad's Power Query merge of ZSDCC + ZSCC (per Tamimi meeting notes 2026-04-30)
WITH zsdcc AS (
    SELECT
        transaction_date,
        store_id,
        COUNT(DISTINCT customer_id) AS customer_count_zsdcc
    FROM {{ source('silver', 'sap_zsdcc') }}
    GROUP BY 1, 2
),
zscc AS (
    SELECT
        transaction_date,
        store_id,
        customer_count AS customer_count_zscc
    FROM {{ source('silver', 'sap_zscc') }}
)
SELECT
    COALESCE(zscc.transaction_date, zsdcc.transaction_date) AS transaction_date,
    COALESCE(zscc.store_id,         zsdcc.store_id)         AS store_id,
    -- Tamimi rule: ZSCC is authoritative when present; fall back to ZSDCC
    COALESCE(zscc.customer_count_zscc, zsdcc.customer_count_zsdcc) AS customer_count,
    CASE
      WHEN zscc.customer_count_zscc IS NOT NULL THEN 'zscc'
      ELSE 'zsdcc'
    END AS customer_count_source
FROM zsdcc
FULL OUTER JOIN zscc USING (transaction_date, store_id)
```


### Singular test — cross-layer reconciliation

```sql
-- tests/assert_unified_sales_reconciles_with_silver.sql
-- Fails if Gold sum diverges from Silver sum for last 7 days by > 0.5%
WITH gold_total AS (
    SELECT
        transaction_date,
        SUM(net_sales) AS sum_gold
    FROM {{ ref('fct_unified_sales') }}
    WHERE transaction_date >= CURRENT_DATE - 7
    GROUP BY 1
),
silver_total AS (
    SELECT
        transaction_date,
        SUM(amount) AS sum_silver
    FROM {{ source('silver', 'ncr_sales') }}
    WHERE transaction_date >= CURRENT_DATE - 7
    GROUP BY 1
)
SELECT
    g.transaction_date,
    g.sum_gold,
    s.sum_silver,
    ABS(g.sum_gold - s.sum_silver) / NULLIF(s.sum_silver, 0) AS drift_pct
FROM gold_total g
    JOIN silver_total s USING (transaction_date)
WHERE ABS(g.sum_gold - s.sum_silver) / NULLIF(s.sum_silver, 0) > 0.005
```

```yaml
# .github/workflows/dbt-test.yml
name: dbt

  pull_request:

permissions:
```

`dbt build` runs:
1. `dbt seed` — load static reference CSVs
2. `dbt snapshot` — SCD2 dimensions
3. `dbt run` — materialise models (incremental Gold facts; full-refresh views)
4. `dbt test` — every model's YAML tests + singular tests

CI fails on first non-pass.

## 11. Patterns

- **Three-layer dbt:** `staging` (views, 1:1 mirror) → `intermediate` (ephemeral, joins/rules) → `marts/core` (tables) + `marts/reporting` (views).
- **Incremental facts, full-refresh dimensions and views.** Dims are small; views are free; facts are the only thing where incremental pays off.
- **`unique_key` + `merge` strategy** on every incremental — never `delete+insert` (causes Redshift VACUUM churn).
- **Audit columns via macro** (`{{ audit_columns() }}`) — `_run_id`, `_ingested_at`, `_dbt_version`.
- **`on_schema_change: fail`** — incremental models break loudly when the schema drifts upstream.
- **`pre_hook: ANALYZE PREDICATE COLUMNS`** on big facts — Redshift planner stats refreshed automatically.
- **Grants in `on-run-end` hook** — Power BI groups always have current grants; no drift.
- **Source freshness checks** — Silver Iceberg must be fresh before Gold runs.
- **Singular tests for cross-layer reconciliation** — Gold sum vs Silver sum tolerance check.
- **`dbt docs serve`** for lineage exploration — operator dashboard can iframe this.

## 12. Anti-patterns

- **Don't COPY-write to Gold tables from outside dbt.** Single owner = dbt.
- **Don't put DAX-style "what if" logic in Gold fact tables.** That's `reporting.vw_*` job.
- **Don't `materialized: table` on reporting views.** Refresh windows + staleness > view cost.
- **Don't query Silver via Spectrum from Power BI.** Go through Redshift views.
- **Don't use `delete+insert` incremental.** MERGE only.
- **Don't snowflake.** Star schema. Dim_store, dim_customer, dim_category are flat.
- **Don't use `dbt run-operation`** for routine grants — put grants in `on-run-end` so they're idempotent.
- **Don't hand-write `CREATE SCHEMA`** — use a macro called in `on-run-start`.
- **Don't commit `profiles.yml`** — only `profiles.template.yml`. CI substitutes secrets via `envsubst`.

## 14. Pitfalls

| Symptom | Cause | Fix |
|---|---|---|
| `dbt run` fails — Silver schema not found | External schema not created on Redshift namespace | `CREATE EXTERNAL SCHEMA silver FROM DATA CATALOG DATABASE 's3tablescatalog/...' IAM_ROLE 'arn:aws:iam::...'` |
| Incremental Gold table missing rows after schema change | `on_schema_change: append_new_columns` silently ignored new business rule | Always `on_schema_change: fail` + explicit migration PR |
| Power BI report sees stale data | Refresh cadence misaligned; `mv_*` not refreshed | Tighten MV refresh schedule; verify `reporting.vw_*` is the model, not a copy |
| `dbt test` extremely slow | Tests scanning full Gold history | Add `where` clauses to test configs; for incremental models, test last N days |
| Redshift workgroup OOM during `dbt build` | RPU max too low; or memory-heavy MERGE | Raise `max_capacity`; profile via `SYS_QUERY_HISTORY` |
| Redshift workgroup runaway cost during `dbt build` | `max_capacity` NOT SET → auto-scales to 512 RPU + no ceiling | **F-AFIE-14:** the workgroup this partial consumes MUST have `max_capacity` set explicitly in its CDK definition (see `MLOPS_DATA_PLATFORM.md` / `DATA_LAKEHOUSE_ICEBERG.md` / `DATA_ZERO_ETL.md`). AFIE Sprint 10 F-FIN-05 retro: $300+/hr burn during a runaway MERGE before someone killed it. |
| dbt + Glue Python shell can't find `dbt-redshift` | Wheel not in `--additional-python-modules` | Bake dbt-core + dbt-redshift into the Glue job's modules list |
| dbt artifacts not visible to operator dashboard | `manifest.json` / `catalog.json` not uploaded post-run | Add S3 upload step at end of Glue dbt job; operator dashboard reads from S3 |

## 15. Acceptance criteria

A dbt project composing this partial passes ALL of:

1. `dbt deps` resolves.
2. `dbt parse` succeeds — no compile-time errors.
3. `dbt source freshness` returns "fresh" for all Silver sources (within configured thresholds).
4. `dbt build --target dev` succeeds — all models built + all tests passed.
5. Every Gold model has `not_null` + `unique` on its PK column.
6. Every reporting view has `select` grants to the Power BI Azure AD groups.
7. CI runs `dbt parse + build` on every PR touching `src/redshift_migrations/`.
8. Prod schedule: `dbt run` + `dbt test` via Glue/MWAA at agreed cadence (typically daily for facts, on-demand for dims).
9. Reconciliation singular test passes — Gold totals match Silver totals within configured drift tolerance.
10. `dbt docs generate` + upload produces a live lineage portal consumed by the operator dashboard.
